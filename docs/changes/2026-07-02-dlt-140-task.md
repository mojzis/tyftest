# 2026-07-02 — new task dlt-140 (incremental dedup-key precedence)

## What changed

- **New task `dlt-140`** from fix `a2e09f28d` (PR #3679, "Fix primary key
  assignment for incremental resources", issue #3657). Same shape as dlt-131:
  a real user-reported precedence bug where a resource's `primary_key` hint
  silently overrides a deduplication key explicitly set on the incremental
  (including the documented `primary_key=()` disable). Mechanism spans
  `extract/incremental/__init__.py`, `extract/incremental/transform.py`,
  `extract/resource.py`, plus removal of the old `sql_database/helpers.py`
  workaround. Validated: gate1 FAIL / gate2 PASS / gate3 PASS → VALID (gate1
  fails on assertions, not collection). Fixtures A–D built. Prompt from the
  issue, symptom-only (file/line hint in the original report stripped).
- **Rejected candidate `4e9ab4ec6`** ("replays state transitions after crash",
  #3767) — right size, but every crash-replay test drives the *new*
  pending-transition API (`save_pending_transition`, `list_pending_transitions`)
  directly, i.e. fully mechanism-coupled (the dlt-131 trap); the only
  behavior-level test in that commit covers an unrelated bundled merge-dedup
  fix, which would let an agent pass without touching the main fix.
- **Gold-set curation:** the fix commit bundles a mypy bump that touched ~26
  extra files with 1–2-line annotation-only edits. `gold_files.txt` is curated
  down to the 4 mechanism files (early-read precision would otherwise be
  measured against annotation noise); `solution.patch` keeps the full source
  diff so gate 2 applies the complete reference fix.
- **Oracle curation:** behavior-only tests via public API
  (`@dlt.resource(primary_key=...)` + `dlt.sources.incremental(...,
  primary_key=...)`, asserting on `unique_hashes` state) — 6 ids covering the
  four setter paths (decorator param default, decorator `incremental` arg,
  `apply_hints(incremental=...)`, late `apply_hints(primary_key=...)`) plus an
  arrow-table variant. Excluded as mechanism-coupled: `test_incremental_merge_
  preserves_dedup_key_flag`, `test_set_hints_*` (assert new internals
  `_dedup_key_from_hints` / call `set_deduplication_key`) and
  `test_dynamic_dedup_key` (callable dedup key = new capability the prompt
  doesn't demand).
- **`launder.sh` OPTIONAL_MODS extended** with `airflow|connectorx`:
  dlt-140's gold files contain runtime-guarded optional-integration imports
  (function-local try/except → MissingDependencyException) that ty flags as
  unresolved-import; without the filter the task was falsely labeled
  `ty_status=degraded`. Not having these installed is the normal state; ty
  still resolves all first-party symbols.
- **New round script `harness/run_dlt140_opus.sh`** (round5, A/D, REPS=5,
  WALL=2400 — between dlt-120's 1200 and dlt-131's 3600).

## Files touched

`holdout/dlt-140/*`, `fixtures/dlt-140/{A,B,C,D}`, `repos/stage-dlt-140/`,
`harness/launder.sh` (OPTIONAL_MODS), `harness/run_dlt140_opus.sh`.

## What it invalidates

Nothing retroactively. The OPTIONAL_MODS change only affects the
`ty_status` label computed at launder time; earlier tasks' labels were
computed without airflow/connectorx imports in their gold files, so they
are unchanged. No venv changes.
