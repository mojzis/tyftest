# 2026-07-01 — new task dlt-131 (bigger than dlt-120) + dlt venv additions

## What changed

- **New task `dlt-131`** from fix `97b3e2e44` (PR #3431, "merge_columns not
  removing compound properties"). Chosen because dlt-120 converged in ~40 steps
  and we want a task closer to real-world 100+-step work: 8 gold source files
  across `common/schema` (schema.py, typing.py, utils.py), `extract`
  (hints.py, extractors.py) and collateral (clickhouse_adapter, rest_api
  config_setup, docs snippet), ~1.2k-insertion fix, 25-iteration upstream PR.
  Validated: gate1 FAIL / gate2 PASS / gate3 PASS → VALID. Fixtures A–D built.
- **Oracle curation constraint discovered:** the commit's big test files
  (`tests/common/schema/test_merges.py`, `tests/pipeline/test_schema_updates.py`)
  parametrize over `ColumnPropInfos[...].compound` at *collection* time — they
  fail to even collect unless the agent reproduces the reference solution's
  exact `TColumnPropInfo.compound` field. That is mechanism-coupled, so those
  files are excluded from BOTH oracle_tests and pass_to_pass. Oracle set is
  behavior-only: `test_apply_hints`, `test_apply_hints_keys[primary_key|merge_key]`
  (extract, unit) and `test_replacing_merge_key[duckdb-no-staging]`
  (end-to-end merge via embedded duckdb — still offline).
- **dlt venv additions** (`repos/dlt.venv`): `pyarrow`, `pandas`, `botocore`
  (module-level imports of the merge_disposition oracle path), `pydantic`
  (optional dlt dep; without it ty reported an unresolved-import in
  `dlt/common/schema/typing.py` → false `ty_status=degraded`).

## Files touched

`holdout/dlt-131/*`, `fixtures/dlt-131/{A,B,C,D}`, `repos/stage-dlt-131/`,
`repos/dlt.venv` (packages only).

## What it invalidates

Nothing retroactively — venv additions are pure additions; earlier tasks'
oracles don't import them. `OFFLINE_DEPS` in `harness/config.sh` now includes
pyarrow/pandas/botocore/pydantic so a from-scratch `setup_repo.sh` rebuild
reproduces the env.
