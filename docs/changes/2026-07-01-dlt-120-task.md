# 2026-07-01 — new held-out task dlt-120 (harder normalizer bug)

## What
Added a third held-out task, **dlt-120**, harder than dlt-110 (round-2 style):
normalizer emits spurious schema updates for nested child tables that end up with
no values (None / empty nested item).

- **fix_sha:** `c8b3da5b5` ("Fix/67 normalizer child table behavior")
- **gold source (4 files, 3 modules):** `dlt/common/normalizers/json/__init__.py`,
  `dlt/common/normalizers/json/relational.py`, `dlt/common/schema/utils.py`,
  `dlt/normalize/items_normalizers.py` (+ `tests/.../custom_normalizers.py`, a test
  helper misclassified into SRC by make_task — §3 quirk, harmless).
- **oracle (FAIL_TO_PASS):** `tests/normalize/test_json_item_inference.py::test_coerce_null_value_in_nested_table`
- **pass_to_pass:** `tests/normalize/test_json_item_inference.py::test_coerce_row`
- Fully offline (drives `JsonLItemsNormalizer` directly; no destination/duckdb).
- Prompt is symptom-only (no file/function/mechanism names).

## Why it's a good "harder" task
4 source files across 3 modules → real cross-module navigation vs dlt-110's 2 files
in one dir, but still a single focused concern. Passes the validity gate cleanly
(no bundling): gate1 FAIL / gate2 PASS / gate3 PASS.

## Harness change (reproducibility)
dlt-120's pre-fix tree (`c8b3da5b5^`, ~dlt 1.15) **hard-imports `hexbytes`** in
`dlt/common/json/__init__.py`. The pin tree (1.28, used by dlt-110) guards that
import, so `hexbytes` was never in `repos/dlt.venv` — `validate.sh` first reported
dlt-120 INVALID on a *collection* error (`ModuleNotFoundError: hexbytes`) that the
gate misread as gate failures.

- Fix: added `hexbytes` to `OFFLINE_DEPS` for dlt in `harness/config.sh` so
  `setup_repo.sh` installs it. Also installed into the existing `repos/dlt.venv`.
- After that, all three gates pass (verified in an isolated venv copy to avoid
  racing the shared venv `.pth` while the sonnet round was running).

## Invalidates
Nothing prior. dlt-110 unaffected (`hexbytes` install is additive). Not yet wired
into any run round; `run_opus_round.sh` / `run_sonnet_round.sh` still target
dlt-110 only.

## Note
`validate.sh` treats any pytest non-zero exit as "test failed", so a missing-dep
collection error masquerades as a gate failure. When a gate result is surprising,
check for import/collection errors before discarding the task.
